## Context

The current trip detail page is generated by `build_trip_detail_html()` in `app/trip_display/rendering.py`. It already has the right data model for a useful lodging review page: trip title, counts, platform filters, availability filters, decision status filters, thumbnails, source links, maps links, Notion links, and decision forms. The main gap is presentation. The current layout reads like a generic dashboard with a gradient hero, rounded cards, pill metrics, and large image cards, while the requested reference at `/Users/ken/Downloads/Trip _.html` reads like a compact trip log or printed itinerary.

Because this repository has no frontend build pipeline and the existing trip display surface is intentionally server-rendered, this restyle should stay inside the HTML renderer and its tests. The implementation should not add React, static asset bundling, or template infrastructure unless later requirements justify it.

## Goals / Non-Goals

**Goals:**
- Match the reference direction: quiet paper background, centered narrow sheet, thin separators, serif title, mono metadata, compact statistics grid, row-based lodging entries, and subdued action links.
- Preserve all existing trip display behavior: filtering, sorting, thumbnail rendering, fallback thumbnails, source/map/Notion links, and lodging decision forms.
- Improve mobile readability by making long lodging names wrap cleanly, keeping controls tappable, and avoiding layout shifts when thumbnails or optional links are missing.
- Keep the page self-contained inside server-rendered HTML without introducing a frontend dependency or external font dependency.
- Add tests that verify the new structure and prevent regressions in key visual hooks and user controls.

**Non-Goals:**
- No new persistence fields, API routes, or LINE command behavior.
- No React/Next/Vite migration.
- No AI summary, itinerary generation, or lodging comparison scoring.
- No pixel-perfect clone of the reference file; the goal is to adapt the style to real `TripDisplaySurface` data and existing controls.
- No image generation or new bitmap asset pipeline.

## Decisions

### 1. Keep a self-contained server-rendered document

The restyle will modify `build_trip_detail_html()` and helper functions in `app/trip_display/rendering.py`. CSS remains inline in the returned HTML string, and the markup remains generated from `TripDisplaySurface`.

Rationale:
- The current app already serves the trip page from FastAPI with no template engine.
- A narrow presentation change does not justify a frontend build stack.
- Keeping the surface self-contained makes tests simple and keeps deployment unchanged.

Alternatives considered:
- Introduce React like the reference HTML.
  - Rejected: it would add runtime weight and a parallel frontend style for a page that is mostly static server-rendered content.
- Add Jinja templates.
  - Rejected for this change: useful later if HTML grows much more complex, but not necessary to deliver this visual restyle.

### 2. Treat the page as a document sheet, not an app dashboard

The top-level layout should use a muted page background and a centered document sheet. Desktop should not be locked to the mobile-sized 600px reading width; it should expand to a wider review surface around 1100px, then step down for tablet and phone breakpoints. The sheet should have a thin dark top border, subtle shadow, square or lightly rounded edges, and simple section dividers. The hero becomes a header block, not a floating card.

Rationale:
- The reference's strongest signal is a printable trip log: one contained page, strong title, fine lines, and dense rows.
- This fits lodging review better than a large dashboard grid because users compare many candidates on a phone.
- Desktop users need enough horizontal space to compare lodging rows without feeling like they are viewing a phone card in the middle of the screen.

Alternatives considered:
- Keep the current 960px card grid and only adjust colors.
  - Rejected: the overall interaction would still read as a dashboard, not as the requested trip log.

### 3. Use row-based lodging entries with stable thumbnail dimensions

Each lodging should render as a bordered row section with:
- a mono row label and platform/status metadata,
- an optional booked/pending/dismissed badge,
- a thumbnail or deterministic fallback on the left,
- title, location/address, captured/updated time, price/availability metadata, and links/actions on the right.

The thumbnail box should keep stable dimensions, such as roughly 108x86 on compact layouts, with responsive constraints for smaller screens.

Rationale:
- The reference uses list rows, not standalone cards.
- Stable thumbnail dimensions prevent layout movement when image URLs are absent or slow.
- The existing `TripDisplayLodging` data maps cleanly to this structure.

Alternatives considered:
- Preserve large 4:3 image cards.
  - Rejected: they reduce scan density and make six to ten candidates harder to compare quickly.

### 4. Convert controls into compact document controls

Filters and sorting should stay available, but the visual weight should be reduced. For desktop, the page should follow the reference information architecture: platform and quick-filter navigation live in a left sidebar, while the right pane keeps a sticky toolbar for availability, decision status, sort, apply, and reset. On tablet and phone widths, the same controls collapse into a stacked flow without losing the same query behavior.

Rationale:
- Users still need filtering and sorting.
- The desktop reference works because context and navigation stay pinned on the left while the list gets most of the horizontal space.
- Controls should not dominate the first viewport more than trip title, counts, and lodging rows.

Alternatives considered:
- Hide filters behind disclosure.
  - Rejected for now: it reduces discoverability and adds interaction complexity without a clear need.

### 5. Use system font stacks and restrained colors

The implementation should use local font stacks: serif for the trip title, sans-serif for body text, and monospace for labels/counts. Colors should stay near the reference: off-white backgrounds, black/dark ink, warm gray dividers, subtle amber warning, and muted secondary links. Avoid gradients, decorative blobs, and saturated single-hue themes.

Rationale:
- External Google Fonts from the reference are not necessary for the product page.
- System fonts keep the page fast and avoid external network dependency.

Alternatives considered:
- Load the exact Google fonts from the reference.
  - Rejected: it introduces network dependency and is not required for the style direction.

## Risks / Trade-offs

- [Inline HTML/CSS string becomes harder to edit] -> Keep helper functions small, use semantic class names, and cover important output with tests.
- [Reference style may reduce visible image size] -> Keep thumbnails inspectable and link them to the source listing when available.
- [Compact layout may crowd decision actions] -> Use wrapping action rows and stable button/link sizes so mobile users can still tap controls.
- [No external fonts means not pixel-identical to reference] -> Preserve the typography hierarchy with system serif/sans/mono stacks instead of exact font files.
- [Tests can become too brittle if they assert full HTML] -> Assert durable markers, labels, class names, and key behavior rather than full snapshots.

## Migration Plan

1. Update `build_trip_detail_html()` CSS and top-level markup to render the document sheet header, stats grid, controls, lodging rows, and footer.
2. Update lodging card helper output to row-based markup with thumbnail/fallback, metadata, links, and decision actions.
3. Ensure empty state, invalid/missing image fallback, missing Notion links, and mobile media queries still render cleanly.
4. Add or update tests in `tests/test_trip_display.py` for key style hooks and preserved controls.
5. Run focused trip display tests, then the full test suite if time allows.

Rollback strategy:
- Revert `app/trip_display/rendering.py` and the associated tests; there are no data migrations or API changes.
